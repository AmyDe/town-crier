# 0022. Offer Codes for Subscription Grants

Date: 2026-04-21

## Status

Accepted

## Context

[ADR 0010](0010-subscription-entitlement-flow.md) established the App Store as the sole path to a paid subscription: the iOS app initiates a StoreKit 2 purchase, the API verifies the resulting JWS, and entitlements are written to Cosmos. That flow is correct for revenue-bearing purchases, but it does not serve several cases that came up as the product matured:

1. **Early supporters and beta testers** — people who helped shape the app and should receive a Personal or Pro tier without paying.
2. **Compensation for service problems** — a user who was affected by an outage or bug needs a tier grant outside the App Store refund flow.
3. **Partner / community promotions** — groups whose members should receive a time-limited Pro tier (e.g., a residents' association, a planning advocacy charity).
4. **Admin testing** — an out-of-band way to put a real user account into a paid tier without going through sandbox purchase flows.

Apple's promotional offer codes partially cover (3) and (4) but require App Store Connect setup per campaign, only work for iOS users, and count against the paid conversion metrics. We also need the same mechanism to grant entitlements to web users once the web app supports authenticated features fully.

## Decision

Introduce an **Offer Code** primitive owned entirely by Town Crier, independent of the App Store.

### Model

An offer code is a short alphanumeric string (generated server-side) that grants a subscription tier for a fixed duration when redeemed by an authenticated user. Data model:

- `OfferCode` domain entity — code string, tier (`personal` / `pro`), duration, optional batch tag, redemption state (unredeemed / redeemed), redeeming user id, redeemed-at timestamp.
- Stored in a new Cosmos `OfferCodes` container partitioned by `/code` (see [ADR 0008](0008-cosmos-db-data-model.md) 2026-04-21 amendment). Point reads on redemption are single-partition.

### Generation and redemption

- **Generate:** `POST /v1/admin/offer-codes` — admin-only endpoint creates a batch of codes using `IOfferCodeGenerator`. Codes are URL-safe, human-readable, and resistant to OCR confusion (no `0`/`O`, `1`/`I/l`).
- **Redeem:** `POST /v1/offer-codes/redeem` — authenticated user submits a code. `RedeemOfferCodeCommandHandler` looks up the code, verifies it is unredeemed, checks the user is not already on a paid tier (`AlreadySubscribedException`), marks it redeemed, and writes the tier + expiry to the user's profile.

### Precedence with App Store entitlements

App Store subscriptions remain the source of truth for App Store-funded tiers. Offer-code grants write to the same `subscriptionTier` / `subscriptionExpiry` fields on the user profile but carry no `originalTransactionId`. If a user later purchases via the App Store, the JWS-verification path overwrites the offer-code grant — App Store state wins.

### Clients

- **iOS** (`town-crier-app/Sources/.../RedeemOfferCodeView`): a settings-screen entry to paste a code, call the API, and surface success or an error toast.
- **Web** (`features/offerCode/`): a redemption hook (`useRedeemOfferCode`) and page with the same behaviour.
- Both use the existing authenticated API client — no additional auth path.

### Not in scope

- Self-service sale of codes to end users. Codes are only generated by admins at the moment.
- Time-bounded or usage-bounded codes beyond the grant duration. A code either grants or does not grant; combinatorial rules are out of scope.
- Integration with Apple's App Store Connect promotional offer codes. Those remain a separate (unimplemented) mechanism if ever needed.

## Consequences

### Easier

- Tier grants work cross-platform (iOS and web) and without depending on App Store Connect workflow.
- Compensation and beta-tester access become a one-liner admin call instead of a support ticket explaining how to use Apple's promotional offer tooling.
- Redemption flow is testable with fakes (`InMemoryOfferCodeRepository`) and has no external dependencies.

### Harder

- Offer codes bypass Apple's revenue reporting and cannot be reported to Apple as promotional offers. This is acceptable because the primary purpose is compensation and beta access, not promotion.
- A redeemed offer code that later expires reverts the user to `free`, even if they had an active App Store subscription in parallel — the profile only stores one tier / expiry pair. Mitigation: the redemption handler rejects codes when the user is already on a paid tier, and App Store `DID_RENEW` notifications overwrite the grant. If Family Sharing or stacked offers are ever added, the model will need revisiting.
- Entitlement code paths must treat an offer-code grant and an App Store grant identically — anything that branches on "how did this user become paid?" creates drift between the two sources.
